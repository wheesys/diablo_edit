# DiabloEdit 总体技术架构

> **最后更新**: 2026-06-15
> **当前版本**: v105 (D2R 3.1, 0x69) 适配完成，全47物品解析通过

## 1. 项目概述

DiabloEdit 是一个从 MFC 迁移到 Qt/C++ 的 Diablo II / Diablo II: Resurrected 存档编辑器。核心功能是**读取、编辑、保存 .d2s 存档文件**，支持从经典 Diablo II (1.13) 到 D2R 3.1 的全部版本。

### 1.1 技术栈

| 层级 | 技术 | 说明 |
|------|------|------|
| UI 框架 | Qt 5/6 | 从 MFC 迁移而来 |
| 语言 | C++17 | `-std=gnu++1z` |
| 构建 | qmake (主) / CMake (工具) | 主项目 Makefile + tools CMake |
| 数据压缩 | QuickLZ 1.5.0 | `.dat` 文件解压 |
| 数据格式 | 自制 TSV | `.txt` 制表分隔文本 → 编译为 `.dat` |

### 2.1 数据层：BinDataStream

`src/core/BinDataStream.h`（单文件，无 .cpp）是整个系统的**底层位流引擎**。

#### 核心类

```
CInBitsStream              COutBitsStream
├── bits_, bytes_          ├── bits_, bytes_
├── CopyBits() 位拷贝       ├── CopyBits() 位拷贝
├── ReadPod<T>() 字节读取   ├── WritePod<T>() 字节写入
├── ReadBits<T>() 位读取    ├── WriteBits<T>() 位写入
├── AlignByte()             ├── AlignByte()
├── SeekBack()              ├── offset_value<> 回填
├── SavePosition()  ← 新增 └── WriteBytes()
└── RestorePosition() ← 新增
```

**关键设计**：
- **`bits<T>(v, b)`** 包装器：声明按 `b` 位读写变量 `v`
- **`>>` / `<<` 运算符重载**：统一语法 `bs >> bits(field, N)` 支持 POD 和位字段
- **`OffsetValue<T>`**：写入时回填机制，用于文件头部写入 Size/CRC（写入时位置未知，写完后回填）
- **`AlignByte()`**：位流指针对齐到下一字节边界（跳过 `(bits_+7)/8` 字节）
- **`SavePosition/RestorePosition`**（v105 新增）：非破坏性 peek 探测

#### 流操作示例

```cpp
// 读取: 2字节 magic + 9-bit index
WORD magic; BYTE hasGolem;
bs >> magic >> bits(hasGolem, 1);     // 1-bit 标志
bs >> bits(value, 10);                 // 10-bit 字段
bs.AlignByte();                        // 字节对齐

// 写入: 回填机制
bs << DWORD(0);                        // 占位
DWORD offSize = bs.BytePos();          // 记录位置
// ... 写入中间数据 ...
bs << offset_value(offSize, newVal);   // 回填到记录位置
```

### 2.2 业务层

#### 2.2.1 存档主结构 (CD2S_Struct)

`src/core/D2S_Struct.h/cpp` — 映射完整的 .d2s 二进制布局：

```
┌─ Header ─────────────────────────────────────────────┐
│ dwMajic(AA55AA55) | dwVersion | dwSize | dwCRC        │
│ dwWeaponSet | Name[16] | charType/Title/Class/Level   │
│ dwSkillKey[16] | dwLeftSkill1/2 | dwRightSkill1/2     │
│ outfit[16] | colors[16] | Town[3] | dwMapSeed         │
│ bMercDead | dwMercControl | wMercName/Type/Exp        │
│ NamePTR[0x94](v105) / NamePTR[0x3C](D2R2.x)           │
├─ Sections ────────────────────────────────────────────┤
│ QuestInfo   → CQuestInfo (3×QuestInfoData, 96B each)  │
│ Waypoints   → CWaypoints (3×WaypointData)             │
│ NPC         → BYTE[0x34]                              │
│ PlayerStats → 0x6667 + 16 stats (9-bit indexed)       │
│ Skills      → 0x6669 + bSkillLevel[30]                │
│ ItemList    → JM(count) + CD2Item[]                   │
│ Corpse      → 0x4D4A + optional CCorpseData           │
│ Mercenary   → 0x666A + optional CItemList             │
│ Golem       → 0x666B + bHasGolem + optional CD2Item   │
│ Trailer     → [TAIL_BRIDGE] + 0x666C + dwUnknown(v105)│
└───────────────────────────────────────────────────────┘
```

#### 2.2.2 物品结构 (CD2Item)

`src/core/D2Item.h/cpp` — 最复杂的子系统（~1400行），实现 `.d2s` 物品位流编解码：

```
CD2Item
├── Header: 32-bit flags (v105) / 27-bit (old)
│   ├── bQuest, bIdentified, bDisabled, bSocketed
│   ├── bNew, bEar, bNewbie, bSimple, bEthereal
│   ├── bPersonalized, bRuneWord
│   └── ext(3-bit v105 / 8-bit old)
├── Layout: iLocation(3) + iPosition(4) + iColumn(4) + iRow(3) +
│   bUNKNOWN_12(1) + iStoredIn(4) = 15 + 1 + 4 bits
├── Huffman code: TYPE_NAME (4 chars, 5-21 bits)
├── CItemInfo (optional, if !bSimple)
│   ├── CExtItemInfo: Gems(3) + GUID(32) + iDropLevel(7) +
│   │   iQuality(4) + bVarGfx/Class/LoQual/HiQual +
│   │   Prefix/Suffix/Set/RareName/Uni/Craft/Rune/
│   │   PersonName/MonsterID/Charm/SpellID
│   ├── bHasRand(1) + optional timestamps
│   └── CTypeSpecificInfo:
│       ├── iDefence(11), iMaxDurability(8), iCurDur(9)
│       ├── iSocket(4) + sub-items (if bSocketed)
│       ├── Quantity flag(1) + iQuantity(9)
│       ├── Set property flags + CPropertyList[]
│       └── RuneWord CPropertyList
├── Sub-items: gem/socket items (if bSocketed)
└── trailing: v105 simple=8/16-bit marker, non-simple=1-bit padding
```

#### 2.2.3 属性系统 (CPropertyList)

```
CPropertyList
├── 读取: 循环读取 9-bit ID
│   ├── ID == 0x1FF → 终止 (terminator)
│   └── ID != 0x1FF → 读取 savebits[ID] 位的值
├── 写入: 同上逻辑
└── CPropertyMetaData 提供每版本每属性的 bits/base/min/max
```

**v105 属性数据修复**：
- `MetaData::findData()` 原始正向迭代返回旧版本数据 → 改为反向迭代 (`rbegin→rend`)，优先匹配 verMin=105
- 从 ItemStatCost.txt 补充 8 个缺失属性 (ID 185/186/205/206/207/213/365/366)

#### 2.2.4 MayExist 模板

`src/core/MayExist.h` — 实现可选字段的懒分配：

```cpp
MayExist<CExtItemInfo> pExtItemInfo;    // 空 → 不存在
pExtItemInfo.ensure()                   // 首次访问时分配
pExtItemInfo.exist()                    // 是否已分配
pExtItemInfo->method()                  // 已分配时访问 (assert)
```

这是 **YAGNI + 内存优化**：物品有大量可选字段，不读取时不分配内存。

### 2.3 数据管理层 (DataManager)

`src/data/DataManager.cpp` — 加载并索引运行时数据：

```
启动流程:
  loadAll(dataPath)
  ├── readLangRes()    → language.dat (QuickLZ解压 → TSV解析)
  ├── readItemRes()    → itemdata.dat (物品元数据，692条)
  ├── readPropRes()    → property.dat (属性定义，519条目)
  └── readNewChar()    → new.dat (空白模板，D2Error静默catch)
```

**数据查询 API**：
- `itemMetaData(typeID)` → `CItemMetaData*` (含 typeCode, equip, simple 等)
- `propertyMetaData(version, id)` → `CPropertyMetaDataItem` (含 bits/base/min/max)
- `propertyDescription(version, id, value)` → `QString` (格式化属性描述)
- 多语言查询宏 `D2_LANG_GETTER`（含边界检查，防 v105 新 Set Item 越界）

## 3. 版本兼容体系

### 3.1 版本定义 (`D2Version.h`)

```cpp
// 版本号采用游戏版本标识符（非语义化版本）
inline BOOL IsD2R(DWORD v)          { return v >= 0x61; }  // D2R 基础
inline BOOL IsPtr24AndAbove(DWORD v){ return v >= 0x62; }  // PTR 2.4
inline BOOL IsPtr25AndAbove(DWORD v){ return v >= 0x63; }  // PTR 2.5
inline BOOL IsRotWAndAbove(DWORD v) { return v >= 0x64; }  // 术士君临
inline BOOL IsPtr31AndAbove(DWORD v){ return v >= 0x69; }  // D2R 3.1

// 有效版本枚举
IsValidVersion() → 0x69, 0x64, 0x63, 0x62, 0x61, 0x60, 0x5C, 0x59, 0x57, 0x47
```

### 3.2 版本适配点分布

| 适配点 | 触发条件 | 影响字段 |
|--------|---------|---------|
| **Waypoints** | `IsD2R && !IsPtr31AndAbove` | unkown + wSize 从 WORD→DWORD (+4B) |
| **NamePTR** | `IsPtr31AndAbove` | 0x3C → 0x94 字节 |
| **NamePTR** | `IsPtr24AndAbove` | Name[16] → NamePTR[0x3C] |
| **NPC** | `IsPtr31AndAbove` | 固定 0x30 → 可变长 (std::search "gf") |
| **NPC** | `IsD2R && !IsPtr31AndAbove` | 固定 0x30 字节 |
| **ItemFlags** | `IsPtr31AndAbove` | 27 → 32 bits |
| **ItemExt** | `IsRotWAndAbove` | 8 → 3 bits (值固定=5) |
| **Quantity** | `IsRotWAndAbove` | 总是 1-bit 标志 (v105: 非堆叠=0) |
| **Timestamps** | `IsRotWAndAbove` | 限制为 3×32 bits |
| **Simple Trail** | `IsPtr31AndAbove` | 8/16-bit marker |
| **Non-simple Padding** | `IsPtr31AndAbove` | AlignByte前 1-bit padding |
| **Golem TAIL_BRIDGE** | `IsPtr31AndAbove` | 0x0100 after item |
| **Merc/Golem** | `isExpansion() \|\| IsPtr31AndAbove` | Nightmare/Hell难度也支持 |
| **CRC 检查** | `IsPtr31AndAbove` | 允许尾部剩余字节（宽松化） |

### 3.3 版本适配模式

采用**内联条件分支**模式而非虚函数/策略模式：

```cpp
// 示例：D2Item ReadData 中的版本适配
if (IsPtr31AndAbove(dwVersion)) {
    bs >> bits(flags, 32);    // v105: 32-bit flags
} else if (IsRotWAndAbove(dwVersion)) {
    bs >> bits(flags, 29);    // v100: 29-bit flags
} else {
    bs >> bits(flags, 27);    // v97: 27-bit flags
}
```

**优势**：单文件修改点集中，无需虚表开销
**劣势**：代码膨胀，条件分支增多（当前尚可接受）

## 4. D2S 存档格式

### 4.1 总体布局

```
Offset  Size    Section           Magic/Marker    Version Conditional
------  ----    -------           ------------    ------------------
0x00    4       Magic             0xAA55AA55      所有版本
0x04    4       Version           0x61~0x69       所有版本
0x08    4       Size              文件总大小      所有版本
0x0C    4       CRC                             所有版本（回填）
0x10    var     Character Header                 所有版本（含 v105 NamePTR[0x94]）
var     var     QuestInfo         "Woof!" (0x216F6F57) 所有版本
var     var     Waypoints         "WS" (0x5357)  所有版本
var     var     NPC               "gf" marker     v105~=42B 动态长度
var     var     PlayerStats       "gf" (0x6667)  所有版本
var     var     Skills            "if" (0x6669)  所有版本
var     var     ItemList          "JM"            所有版本
var     var     Corpse            "JM"            所有版本
var     var     Mercenary         "jf" (0x666A)  isExpansion||v105
var     var     Golem             "kf" (0x666B)  isExpansion||v105
var     2+4     Trailer           0x666C+dword    v105 only (含TAIL_BRIDGE)
```

### 4.2 v105 关键格式差异

| # | 差异项 | 旧行为 | v105 行为 | 修复提交 |
|---|--------|--------|----------|---------|
| 1 | 物品标志位 | 27 bits | **32 bits** | `5de7755` |
| 2 | D2R 扩展位 | 8 bits | **3 bits** (=5) | `5de7755` |
| 3 | 行字段 | 4 bits | **3+1 bits** (row+unknown) | `5de7755` |
| 4 | 数量标志 | 仅堆叠物品 | **总是 1-bit** | `5de7755` |
| 5 | Simple 物品 trailing | 无 | **16-bit WORD=2** (非socket) | `54413cc` |
| 6 | Socket filler trailing | 16-bit WORD | **8-bit BYTE=2** | `54413cc` |
| 7 | Socket filler 2-bit padding | 无 | **BitPos∈{0,7}时补2-bit** | `bdcc3d1` |
| 8 | Non-simple 1-bit padding | 无 | **BitPos==0时补1-bit** | `be39b58` |
| 9 | Golem TAIL_BRIDGE | 无 | **0x0100 between item & 0x666C** | `eebd9d6` |
| 10 | NPC 段长度 | 固定 0x30 | **可变长 (~42B)** | `dbc2aaf` |
| 11 | bHasRand 位置 | 所有物品 | **仅在 non-simple** | `f3c67ad` |
| 12 | NamePTR 大小 | 0x3C | **0x94** | (初始适配) |
| 13 | Property verMin=105 | 旧条目优先 | **反向迭代匹配** | `MetaData.cpp` |

### 4.3 v105 物品位流格式总结

```
CD2Item::ReadData (v105):
┌─ Header ────────────────────────────────────────┐
│ [32-bit flags] [3-bit ext(=5)]                   │
│ [3-bit location] [4-bit position]                │
│ [4-bit column] [3-bit row] [1-bit unknown]       │
│ [4-bit stored_in]                                │
├─ Huffman Code (≤21 bits) ────────────────────────┤
│ TYPE_NAME (4 chars decoded via HUFFMAN tree)     │
├─ if bSimple ──────────────────────────────────── │
│  [skip body]                                     │
│  [8-bit marker (socket, =2) / 16-bit (others)]   │
├─ if !bSimple ─────────────────────────────────── │
│  [CExtItemInfo] [bHasRand + timestamps]          │
│  [CTypeSpecificInfo] (attributes + CPropertyList)│
│  [if bSocketed: for each gem: Sub-CItemInfo]     │
│  [if BitPos==0: 1-bit padding]                   │
│  [AlignByte]                                     │
├─ Sub-items ───────────────────────────────────── │
│  if bSocketed: CD2Item per gem                   │
└──────────────────────────────────────────────────┘
```

## 5. Huffman 编码

### 5.1 编码表

物品的 4 字符 TYPE_CODE 使用 Huffman 编码压缩，编码表硬编码在 `src/core/D2Item.cpp:42-58`：

```cpp
static const BYTE HUFFMAN[] = {
    // 序列化二叉树，37个字符终端节点
    // 每个终端节点: [字符] (1 byte)
    // 每个内部节点: [右子偏移] [左子偏移]
};
```

**重要发现**：D2R 3.1 (v105) 的 Huffman 表与 D2R 2.x (v97-v99)**完全相同**，无需修改。

### 5.2 字符集

```
0-9, a-z, space → 37个字符，编码为 5-21 bits 不等
```

常见物品码编码长度：`mau`=5bit, `rin`=10bit, `jew`=12bit

## 6. 属性系统

### 6.1 property.txt 格式

```
ID\tVerMin\tFieldBits\tFieldBase\tFieldMin\tFieldMax\t...\tDefaultValue\n
```

**CPropertyField** 结构（每 4 个整数一组）：

| 字段 | 说明 |
|------|------|
| `bits` | 位宽 (savebits) |
| `base` | 基值偏移 |
| `min` | 最小值 |
| `max` | 最大值 |

`DefaultValue` = 最后一个制表分隔整数（C++ parser 从行末读取）。

### 6.2 复合属性 (np grouped stats)

部分属性由多个子值组成（如 ID 54 = coldmindam + coldmaxdam + coldlength）。`bitsSum_` 自动累加所有字段位宽，C++ 代码通过 `MetaData::Parse()` 一次性读取并拆分为多个参数。

### 6.3 propertyDescription 特殊处理

`DataManager::propertyDescription()` 包含 **30+ 个特殊 case**（按 ID 分发），处理单位转换、百分比计算、技能名映射等：

| ID 范围 | 特殊处理 |
|---------|---------|
| 54, 56-57, 59 | 除以 25 |
| 97, 151 | 拼接技能名 |
| 107 | 拼接技能名+职业名 |
| 214-250 | `/8` 浮点格式化 |
| 268-303 | 时间格式化 (timeName) |

## 10. 后续开发建议

### 10.1 短期（高优先级）

1. **Round-trip 验证**：完整读→写→读循环，确保保存不损坏存档
2. **游戏内验证**：在 D2R 3.1 加载编辑后的存档
3. **WriteData 一致性**：逐项对比 WriteData 与 ReadData 的对称性

### 10.2 中期（中优先级）

4. **GUI 功能测试**：角色信息、属性、技能、物品、任务、传送点读取
5. **编辑功能测试**：修改属性/技能/物品后保存验证
6. **补充测试存档**：收集 2.4/2.6/术士君临/经典版存档用于回归测试

### 10.3 长期（低优先级）

7. **ItemGridWidget 拆分**（~1000行→多个子组件）
8. **D2Item.cpp 拆分**（~1400行→按职责分离）
9. **自动化测试框架**：至少验证已知存档可解析
10. **写路径策略重构**：ReadData 和 WriteData 代码重复度高（每个适配点需同步修改两处），可考虑模板化或宏化

### 10.4 架构优化方向

| 方向 | 当前问题 | 建议方案 |
|------|---------|---------|
| 版本适配 | `if(isV31)/else if` 散落各处 | 集中版本差异表，运行时查表 |
| Read/Write 对称 | 每次修改需同步两处 | 静态断言 + round-trip 自动化测试 |
| 错误处理 | D2Error(6) 含义不明确 | 细分错误码，附加上下文（文件偏移） |
| 数据隔离 | DataManager 全局 `g_dataMgr` | 改为依赖注入或 Context 传递 |
