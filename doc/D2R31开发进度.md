## 2026-06-13：Huffman 编码表不兼容 — 分析记录

### 🔴 2026-06-13 晚：证伪 — Huffman 表未变！

通过搜索公开资料（d07RiV, d2r-horadric-tools, D2SLib），发现 **D2R 3.1 (v105) 的 Huffman 编码表与 D2R 2.x (v97-v99) 完全相同**。

**验证方法**：将 `d2r-horadric-tools/d2r_chargen/data/huffman.py`（d07RiV 发布的 D2R Huffman 表，被 v105 项目直接使用）与 `src/core/D2Item.cpp:42-58` 的 HUFFMAN[] 做逐字符对比，所有 37 个字符的 (code, bits) 完全一致。

### ✅ 真正的问题：物品头部字段布局变化

#### 公开资料对比

| 字段 | v97 (D2R 2.x, d07RiV) | v105 (D2R 3.1, d2r-horadric-tools) | 当前 C++ 代码 |
|------|----------------------|-----------------------------------|-------------|
| **物品标志位** | 27 bits | 32 bits (新增 bit 23=1) | 29 bits |
| **D2R 扩展位** | 8 bits | **3 bits (=5)** | 8 bits |
| **行字段 (row)** | 3 bits + 1 未知 bit | 3 bits + 1 未知 bit | **4 bits** |
| **Huffman 前总计** | 27+8+18=53 bits | 32+3+18=**53 bits** | 29+8+18=**55 bits** |

#### 问题根源

C++ 代码在 Huffman 编码前多读了 **2 bits**（29 vs 32 flag bits, 8 vs 3 extension bits, 4 vs 3 row bits），导致比特流偏移。由于每个物品结尾有 `AlignByte()`，偏移量在物品间累积/漂移，使得前几个物品偶尔能正确解析（偶然对齐），后续物品则解码失败。

#### 关键参考源

- **d07RiV D2R 格式说明**: https://github.com/d07RiV/d07riv.github.io/blob/master/d2r.html
  - JS 实现，定义 v97 物品格式
  - `huffman` 嵌套数组 = 与 C++ HUFFMAN[] 相同的树

- **d2r-horadric-tools (v105/RotW)**: https://github.com/crabsmadethis/d2r-horadric-tools
  - Python 实现，完整支持 v105 格式
  - `d2r_chargen/data/huffman.py` — HUFFMAN 字典 (code, bits)
  - `d2r_chargen/build_lib.py` — 权威 v105 物品编码器
  - `docs/d2s_format.md` — 完整的 v105 格式文档

- **D2SLib (C#)**: https://github.com/dschu012/D2SLib
  - 最广泛使用的 D2R 存档库

- **halbu (Rust)**: https://github.com/feored/halbu
  - 新版 D2R 存档库，v105 支持，但暂不处理物品

### 🔧 修复方案

需要修改 `src/core/D2Item.cpp` 的 `CD2Item::ReadData()`:

1. **标志位**: 从 27 bits 扩展到 **32 bits**
2. **D2R 扩展位**: 从 8 bits 减少到 **3 bits**，v105 中值固定为 5
3. **行字段**: 从 4 bits 改为 **3 bits**，之后加 1 bit 未知字段
4. **数量字段**: v105 总是有 1-bit 标志位（非堆叠物品值为 0）

同步修改 `WriteData()` 以实现正确的 round-trip。

### Scanner 检测原理

Scanner 使用**字节模式扫描**（byte-pattern matching）定位物品边界：
```python
for pos in range(merc_jm+4, merc_end-4):
    b0 = data[pos]
    b2 = data[pos+2]
    if b0 == 0x10 and b2 in (0x80, 0xa0, 0xc0, 0xe0):
        # 0x10 = bSimple(bit0) + bSocketed(bit4) + ext bits
        # 0x80/a0/c0/e0 = quality bits 在 byte 2 高位
        decode_item_header(data, pos)
```

以此检测到的 merc item 位置：
| Item | Byte | Dword | 说明 |
|------|------|-------|------|
| vgl | 0x07E2 | 0x00800010 | |
| rin | **0x0809** | 0x00800010 | scanner 定位 |
| amu | 0x0836 | 0x00800010 | |

### 🔴 根本原因：1-byte gap

**C++ 顺序解析器**从 item 0 开始逐字段读取，在 `CD2Item::ReadData` 末尾执行 `bs.AlignByte()` 后，下一个 item 起始位置为 **0x0808**。但实际 item 1 (`rin`) 的 header dword 位于 **0x0809**。

**Python 位级仿真验证**：

```python
def parse_item(start_byte, label):
    # CD2Item header: 32-bit flags + 3-bit ext + 18-bit layout
    dword = struct.unpack('<I', data[start:start+4])[0]
    # ... parse flags, location, position, column, row, stored_in
    # CItemInfo: Huffman decode 4 chars
    # CExtItemInfo: nGems(3)+GUID(32)+ilvl(7)+quality(4)+...
    # CTypeSpecificInfo::ReadData:
    #   iDefence(11) + iMaxDurability(8) + iCurDur(9, if max>0)
    #   + bHasQuantity(1) + iQuantity(9, if true) + IsSet flags
    #   + CPropertyList (9-bit IDs until 0x1FF)

# Item 0 (vgl): 从 0x07E2 开始，hasDef=1, hasDur=1
parse_item(0x07E2, "vgl") → end=0x0808 ✅

# Item 1:
# C++ 位置 0x0808: dword=0x80001000 → code='wh' ❌
# Scanner 位置 0x0809: dword=0x00800010 → code='rin' ✅
```

### 假说：v105 non-simple 物品需要 trailing marker

- **Simple 物品**：已知在 Huffman 码后有 trailing marker（非 socket=16-bit WORD=2，socket filler=8-bit BYTE=2）
- **Non-simple 物品**：当前 C++ 代码在 property list 结束后直接 `AlignByte()`，**未读取任何 trailing marker**

**证据**：
- Item 0 (`vgl`) 末尾字节 `0x0808` 值为 `0x00`
- 如果此字节作为 v105 non-simple 物品的 trailing marker 被消耗，C++ 顺序解析器就能对齐到正确位置 `0x0809`

## 2026-06-15：v105 non-simple 物品 1-bit padding 修复

### 根因分析

参考 [d2r-horadric-tools/build_lib.py](https://github.com/crabsmadethis/d2r-horadric-tools/blob/main/d2r_chargen/build_lib.py#L893-L904):

```python
# D2R v105 requires at least 1 bit of padding after the last terminator.
# Items with exactly byte-aligned data (0 padding bits) cause
# "FAILED TO JOIN GAME" — confirmed by testing sB=3 stats.
if w.pos % 8 == 0:
    w.write_bits(0, 1)
w.align()
```

v105 格式在 `AlignByte()` 之前要求**至少 1-bit padding**。若属性列表终止符（0x1FF）后恰好字节对齐，写入方额外插入 1-bit padding 再 AlignByte，导致读取方多消费 8 bits = 1 字节。

### 修复内容

| 文件 | 变更 | 说明 |
|------|------|------|
| `src/core/BinDataStream.h` | COutBitsStream 添加 `BitPos()` | 写入侧 bit 位置查询 |
| `src/core/D2Item.cpp` ReadData | v105 non-simple padding 读取 | `BitPos==0` 时消费 1-bit |
| `src/core/D2Item.cpp` WriteData | v105 non-simple padding 写入 | `BitPos==0` 时补写 1-bit (=0) |

### 验证结果

**Merc 段独立测试** (9/9 ✅):
```
Merc[0] vgl → Merc[1] rin → Merc[2] amu → Merc[3] rin

```

### v105 Golem TAIL_BRIDGE 格式修复

#### 格式分析

## 2026-06-17：杜五.d2s（1962字节版）物品解析修复

### 背景

用户报告 Linux 下打开 `/opt/diablo_edit/杜五.d2s`（version=0x69, 31 char items, 0 merc）报错 `Failed to load file`，异常消息为空。该文件与已验证的 `杜五_v3.1.d2s`（2425字节, 37+9+1 物品）是不同存档。

### 诊断

通过命令行诊断程序（qmake6 编译核心库 + 诊断 main，绕过 GUI/无 cmake 环境）定位：

- 文件本身完全有效：签名/CRC/所有 section magic（QuestInfo@403 / Waypoints@701 / PlayerStats@833 / Skills@884 / ItemList@916 / stCorpse@1943 / stMercenary@1947 / stGolem@1953）位置全部自洽
- 失败在物品解析：item[0-5] 成功，item[6] 解析后 `bs.Good()==false`
- item[6] @byte 1178 是 simple 物品（code=`rvs`），Huffman 码后停在 byte 1186 **bit 7（未字节对齐）**

### 根因：simple 物品 trailing marker 字节对齐

`CD2Item::ReadData` 用 `bs >> trailingMarker`（`readPod` 读 WORD）读 simple 物品 non-socket 的 16-bit trailing marker，但 `readPod` 要求字节对齐（`ensure` 检查 `bits_==0`）。Huffman 码后流常未对齐，导致 `ensure` 失败、`bad_=true`，后续 item 全部错位，最终 `CCorpse` 读不到 `0x4D4A` 抛 `D2Error(19)`（msgId 越界 → 空消息）。

`杜五_v3.1.d2s` 的 simple 物品恰好字节对齐故侥幸通过；`杜五.d2s` item[6] 的 Huffman 码长度导致未对齐而暴露 bug。

### 修复（提交 `4fd81c2`）

| 文件 | 变更 | 说明 |
|------|------|------|
| `D2Item.cpp` ReadData | `bs >> trailingMarker` → `bs >> bits(trailingMarker, 16)` | 位读取，适应任意对齐 |
| `D2Item.cpp` WriteData | `bs << WORD(2)` → `bs << bits<WORD>(2, 16)` | 位写入，round-trip 一致 |

验证：byte 1186 bit7 位读 16 bits = 2（trailing marker）✓，与 d2r-horadric-tools `write_bits(2,16)` 一致。
