# D2R 3.1 支持 — 数据文件升级方案

## 背景

当前项目已通过 `D2Version.h` 的 `0x69` 版本号识别 D2R 3.1，但 `data/` 目录下的运行时数据文件仍是 **2.5/2.6 版本**。

## 数据文件架构

### 编辑器运行时加载的数据（`data/` 目录）

| 文件 | 格式 | 作用 | 能否从 global/excel/ 推导 |
|---|---|---|---|
| `language.txt` → `language.dat` | 自定义 tab-separated | 所有显示文本（菜单、物品名、技能名） | ❌ 需要 D2R string table |
| `itemdata.txt` → `itemdata.dat` | 自定义 tab-separated | 物品类型元数据（4-byte code → Pic/Equip/Dmg） | ✅ 大部分可推导 |
| `property.txt` → `property.dat` | 自定义 tab-separated | 存档属性 bit 编码规则（每个属性占几bit、偏移量） | ❌ 需要逆向工程 |
| `new.dat` | 原始二进制 | 新建角色模板 | ❌ 需要初始存档 |

### ⚠️ 核心概念区分：property.txt ≠ 物品属性

容易混淆的两个东西：

| 文件 | 本质 | 控制什么 | 3.1 新增物品是否影响 |
|---|---|---|---|
| **`data/property.txt`** | **存档二进制解码表** | 每个属性ID在 .d2s 文件中占几bit、base偏移是多少 | **不影响** — 新物品仍用已有属性ID |
| **`data/itemdata.txt`** | **物品类型注册表** | 4字节code对应哪个Pic索引、装备位置、伤害范围 | **影响** — 新物品有新的code，需要注册 |
| **`global/excel/properties.txt`** | **游戏属性定义表** | 属性名称、关联stat、功能函数 | 不直接用于编辑器 |

**为什么 property.txt 大概率不需要更新：**

3.1 游戏定义的属性共 **284 个**（ID 0-283），而编辑器 `property.txt` 已定义了 **365 个条目**（ID 0-365+），不仅全覆盖还多预留了 80+ 个槽位。

新装备依然使用已有属性（防御、抗性、技能加成等），只是换了个武器模板。只有**全新类型的附魔/属性**（比如 D2R 新增了某种之前完全不存在的 stat）才需要扩展 property.txt，这种情况在 D2R 小版本更新中极少发生。

### 游戏原始数据表（`doc/source/txt/3.1/`）

已归档 180 个文件（90 root excel/ + 90 base/），目录结构与 D2R 官方一致。

其中可用于辅助升级的字段关系：
- `armor.txt` / `weapons.txt` / `misc.txt` 的 `code` 列 → 定义全部 4 字节物品类型码
- `properties.txt` 的 `*Id` 列 → 属性编号（与 `data/property.txt` 的 ID 对应）
- `skills.txt` 的 `skill` 列 → 技能名称

## 升级路线

### 1. `itemdata.txt` — ✅ 可自动生成

**方法**：从 3.1 的 `armor.txt`、`weapons.txt`、`misc.txt` 提取所有 `code` 列，交叉比对 2.5 的 `itemdata.txt`，自动生成更新后的版本。

**需要提供**：`doc/source/txt/3.1/global/excel/` 目录已有 ✅

### 2. `new.dat` — ✅ 可自动生成

**方法**：取一个 D2R 3.1 初始角色存档文件即可。

**需要提供**：D2R 3.1 初始存档（新建角色后保存即可获得）

### 3. `property.txt` — ⚠️ 需要对比分析

**方法**：对比同一角色在 2.5 和 3.1 下的存档二进制差异，识别新增/变化的属性编码规则。

**需要提供**：同一角色的 **2.5 存档** 和 **3.1 存档**

### 4. `language.txt` — ⚠️ 部分需要

**方法**：
- UI 字符串（菜单、对话框）原有即可用，无需更新
- 如果 3.1 新增了物品/技能/属性名，需在对应 section 追加
- 可以从 3.1 存档中提取新出现的字符串索引
