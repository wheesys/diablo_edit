# Diablo Edit 跨平台重写可行性分析报告

## 一、项目现状概览

### 1.1 当前项目架构
- **语言**: C++ (MFC/Win32)
- **UI框架**: MFC (Microsoft Foundation Classes)
- **架构模式**: SDI (单文档界面) + Document/View
- **构建系统**: Visual Studio 2017+ (.sln/.vcxproj)
- **支持版本**: D2 经典 1.00 ~ D2R 3.1 (0x69)，共10个版本
- **源文件**: 约25个核心源文件 + 322+物品位图资源
- **代码量估算**: 约8,000-10,000行C++代码

### 1.2 核心功能列表
| 功能模块 | 对话框 | 复杂度 |
|---------|--------|--------|
| 角色基本信息编辑 | CDlgCharBasicInfo | 中 |
| 技能编辑 (30技能/职业) | CDlgSkills | 中 |
| 物品网格管理 (拖拽) | CDlgCharItems | **极高** |
| 物品属性编辑器 (工坊) | CDlgFoundry | **高** |
| 新物品创建 | CDlgNewItem | 中 |
| 任务状态编辑 | CDlgQuestInfo | 低 |
| 传送点编辑 | CDlgWayPoints | 低 |
| 物品悬浮提示 | CDlgSuspend | 中 |
| 角色选择 | CDlgSelectChar | 低 |

## 二、代码可移植性分层分析

### 2.1 完全可移植层（可直接复用，约35%代码量）

| 文件 | 功能 | 移植难度 |
|------|------|---------|
| `D2S_Struct.h/cpp` | D2S存档结构定义与读写 | ⭐ 几乎零修改 |
| `D2Version.h` | 版本检测函数 | ⭐ 零修改 |
| `BinDataStream.h/cpp` | 位级I/O流 | ⭐ 几乎零修改 |
| `MayExist.h` | 可选字段模板 | ⭐ 零修改 |
| `MetaData.h/cpp` | 属性元数据解析 | ⭐⭐ 小幅修改 |
| `D2Item.h/cpp` | 物品数据结构与序列化 | ⭐⭐ 小幅修改 |

**关键发现**: 核心的D2S文件解析逻辑（位级流、结构序列化、物品系统）与MFC/Win32**完全解耦**，仅使用标准C++类型（BYTE, WORD, DWORD, std::vector等）。这是项目最优良的设计特征，为跨平台重写提供了坚实基础。

### 2.2 需要适配层（中等工作量，约15%代码量）

| 文件 | 功能 | 适配内容 |
|------|------|---------|
| `Diablo Edit2.h/cpp` | 应用主类、数据加载、文本查找 | 替换MFC的CWinApp、资源加载、字符串表 |
| `Generate Data/*` | 数据生成工具 | 替换MFC文件操作为标准C++或平台抽象 |
| QuickLZ压缩库 | 数据压缩/解压 | 纯C实现，零修改 |

### 2.3 完全重写层（主要工作量，约50%代码量）

| 文件 | 功能 | 重写原因 |
|------|------|---------|
| `CharacterDialogBase.h/cpp` | 对话框基类 | MFC CDialog → 跨平台UI框架 |
| `DlgCharBasicInfo.h/cpp` | 角色信息编辑 | MFC控件绑定 → 跨平台UI |
| `DlgCharItems.h/cpp` | **物品网格编辑器** | MFC自绘+拖拽 → 跨平台自绘组件 |
| `DlgFoundry.h/cpp` | **物品属性编辑器** | MFC复杂表单 → 跨平台表单 |
| `DlgSkills.h/cpp` | 技能编辑 | MFC对话框 → 跨平台UI |
| `DlgNewItem.h/cpp` | 物品创建 | MFC树控件 → 跨平台树控件 |
| `DlgQuestInfo.h/cpp` | 任务编辑 | MFC复选框组 → 跨平台UI |
| `DlgWayPoints.h/cpp` | 传送点编辑 | MFC复选框组 → 跨平台UI |
| `DlgSuspend.h/cpp` | 悬浮提示 | Win32分层窗口 → 跨平台透明窗口 |
| `MainFrm.h/cpp` | 主框架 | MFC CFrameWnd → 跨平台主窗口 |
| `DiabloEdit2View.h/cpp` | 主视图 | MFC CFormView → 跨平台视图 |
| `resource.h / .rc` | 资源定义 | Win32资源 → 跨平台资源系统 |

## 三、跨平台方案对比

### 3.1 方案A：Qt (C++)

**推荐指数: ⭐⭐⭐⭐⭐**

| 维度 | 评估 |
|------|------|
| **语言兼容性** | 同为C++，核心逻辑可直接复用 |
| **UI能力** | 完善的控件库、自绘能力、拖放支持 |
| **平台覆盖** | Windows / macOS / Linux |
| **开发效率** | Qt Designer可视化布局，信号槽机制 |
| **生态** | 成熟稳定，文档丰富，社区庞大 |
| **分发体积** | 静态链接约15-25MB，动态链接需附带Qt库 |
| **许可证** | LGPL v3 / 商业许可 |

**关键优势**：
- 核心D2S解析代码（~3500行）可直接复用，无需翻译
- Qt的`QGraphicsView`/`QGraphicsScene`非常适合实现物品网格编辑器
- Qt的拖放框架（Drag & Drop）可完美替代MFC的拖拽实现
- Qt支持资源系统（.qrc），可替代Win32资源文件
- `QTabWidget`直接对应MFC的`CTabCtrl`
- `QTreeWidget`对应`CTreeCtrl`
- 自绘通过`QPainter`实现，比MFC的`CPaintDC`更现代化
- 半透明浮窗通过`Qt::WindowStaysOnTopHint | Qt::FramelessWindowHint`+`setWindowOpacity`实现

**核心代码复用率**: ~70%（逻辑层100%复用 + UI层完全重写）

**预估工作量**: 2-3个月（1人全职）

### 3.2 方案B：Electron / Tauri (Web前端)

**推荐指数: ⭐⭐⭐**

| 维度 | 评估 |
|------|------|
| **语言兼容性** | 需要将C++核心逻辑翻译为JS/TS或通过FFI调用 |
| **UI能力** | HTML/CSS自由度极高，适合复杂表单 |
| **平台覆盖** | Windows / macOS / Linux |
| **开发效率** | Web生态丰富，迭代快 |
| **分发体积** | Electron ~100MB+，Tauri ~10MB |
| **许可证** | MIT |

**挑战**：
- C++核心逻辑需要翻译（约3500行）或通过WebAssembly/N-API桥接
- 位级二进制操作在JS中不如C++自然（需要使用DataView/ArrayBuffer）
- 物品网格拖拽用Web实现反而更方便（HTML5 Drag & Drop）
- Tauri（Rust后端）需要将C++翻译为Rust或通过FFI调用

**核心代码复用率**: ~30%（需翻译核心逻辑）或 ~70%（通过WASM/FFI桥接）

**预估工作量**: 2-4个月（1人全职）

### 3.3 方案C：wxWidgets (C++)

**推荐指数: ⭐⭐⭐**

| 维度 | 评估 |
|------|------|
| **语言兼容性** | 同为C++，核心逻辑可直接复用 |
| **UI能力** | 原生控件，但自绘和拖拽不如Qt方便 |
| **平台覆盖** | Windows / macOS / Linux |
| **开发效率** | 无可视化设计器（官方），API风格接近MFC |
| **分发体积** | 较小，约5-10MB |
| **许可证** | wxWindows Library License (类LGPL) |

**挑战**：
- API风格与MFC类似，迁移学习成本低，但也意味着需要处理类似的底层细节
- 自绘能力不如Qt的QPainter方便
- 物品网格编辑器的实现难度高于Qt
- 社区和文档不如Qt丰富

**核心代码复用率**: ~70%

**预估工作量**: 3-4个月（1人全职）

### 3.4 方案D：Dear ImGui (C++)

**推荐指数: ⭐⭐**

| 维度 | 评估 |
|------|------|
| **语言兼容性** | C++，核心逻辑直接复用 |
| **UI能力** | 即时模式GUI，适合工具类应用 |
| **平台覆盖** | 依赖后端（SDL/GLFW等） |
| **开发效率** | 复杂布局需要手动编码，无设计器 |
| **分发体积** | 非常小，<5MB |

**挑战**：
- 即时模式GUI不适合复杂表单（物品属性编辑器）
- 没有内置的树控件、标签页等高级控件（需要第三方扩展）
- 拖放实现需要大量自定义代码
- 不适合非技术用户使用

**核心代码复用率**: ~70%

**预估工作量**: 4-5个月（1人全职）

## 七、结论

### 7.1 可行性结论：✅ 完全可行

**核心理由**：

1. **核心逻辑与UI完全解耦** — 项目最优良的设计特征。D2S文件解析、物品序列化、版本检测等核心逻辑（约3500行）使用纯标准C++编写，零MFC依赖，可直接在Qt项目中复用。

2. **数据文件格式通用** — language.dat / itemdata.dat / property.dat 使用QuickLZ压缩，QuickLZ为纯C实现，跨平台零障碍。

3. **Qt完美匹配需求** — Qt提供了MFC所有对应的控件和能力，且在自绘、拖放、国际化方面更现代化。

4. **3.1版本支持已成熟** — 当前项目已完整实现3.1版本的存档结构处理，跨平台版本可直接继承。

### 7.2 推荐实施路径

1. **阶段一**（2周）：搭建Qt项目框架，移植Core层，验证D2S文件读写
2. **阶段二**（3周）：实现角色基本信息、技能、任务、传送点编辑
3. **阶段三**（5周）：实现物品网格编辑器（最复杂模块）
4. **阶段四**（3周）：实现物品属性编辑器和物品创建
5. **阶段五**（2周）：多语言、资源集成、测试、打包分发

### 7.3 如果选择其他语言

如果倾向于使用Web技术栈，推荐 **Tauri + React/Vue**：
- Rust后端处理文件I/O和D2S解析（需将C++翻译为Rust）
- Web前端负责UI（HTML/CSS实现复杂表单更灵活）
- 分发体积小（~10MB），性能接近原生
- 但核心逻辑需要翻译，工作量增加约30-50%
